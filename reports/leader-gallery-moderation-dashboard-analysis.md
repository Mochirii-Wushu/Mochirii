# Leader Gallery Moderation Dashboard Analysis

## Current Upload Architecture

Mochirii is a static HTML/CSS/vanilla JavaScript site that runs on GitHub Pages without a build step. Shared runtime code loads in page templates with `utils.js`, the Supabase JS v2 CDN, `supabase.js`, `site.js`, and then page-specific scripts.

Discord-role-gated gallery uploads are implemented through `gallery-submit.html`, `gallery-submit.js`, and shared helpers in `supabase.js`. Authenticated users upload image files into the private `member-gallery` Storage bucket under a path beginning with their `auth.uid()`. The browser creates a `gallery_submissions` row after the Storage upload succeeds. Client-side checks are user experience only; Storage policies and table RLS remain the enforcement layer.

The current member upload limit is 50 MB. Accepted upload MIME types are still `image/jpeg`, `image/png`, and `image/webp`.

## Current Auth And Role Verification Architecture

Supabase Auth uses Discord OAuth through the browser client in `supabase.js`. Discord login proves identity only. Guild membership and role checks are performed server-side in `supabase/functions/verify-discord-member/index.ts` using a Discord bot token stored as a Supabase Edge Function secret.

The existing upload verification checks Discord server `1078630751077142608` and requires both member role IDs:

- `1468659807736299520`
- `1078630751077142615`

The Edge Function writes verified Discord identity and role state into `public.member_profiles`. Browser code does not receive bot tokens, service-role keys, or Discord client secrets.

## Pending Submissions Schema

`public.gallery_submissions` stores member upload review state. The relevant fields for moderation are:

- `id`
- `user_id`
- `storage_bucket`
- `storage_path`
- `original_filename`
- `mime_type`
- `size_bytes`
- `title`
- `caption`
- `category`
- `status`
- `rejection_reason`
- `reviewed_by`
- `reviewed_at`
- `created_at`
- `updated_at`

Valid submission statuses are `pending`, `approved`, `rejected`, and `archived`. Existing browser grants only allow authenticated users to update safe metadata fields on their own pending submissions. Regular browser clients cannot update `status`, `reviewed_by`, `reviewed_at`, or `rejection_reason`.

## Storage Access Model

The `member-gallery` bucket is private. Existing Storage policies require authenticated active verified members to write only under their own first path segment. Authenticated users can read their own member-gallery objects. There is no anonymous or public read access.

Moderation previews must therefore be generated by a privileged Edge Function as short-lived signed URLs. The bucket must remain private, and approved images must not be published to the public gallery in this phase.

## Proposed Edge Functions

The moderation workflow will add:

- `supabase/functions/list-gallery-review-queue/index.ts`
- `supabase/functions/moderate-gallery-submission/index.ts`
- shared moderation helper code under `supabase/functions/_shared/`

Both functions will require a signed-in Supabase JWT and will verify the caller's Discord server membership plus Moderator role ID `1078630751165222984` before performing privileged work.

`list-gallery-review-queue` will return pending submissions with safe uploader display fields and short-lived signed preview URLs for private Storage objects.

`moderate-gallery-submission` will approve or reject only pending submissions, update privileged review fields, and insert an audit row into `public.gallery_moderation_events`.

## Proposed Database Migration

A new migration will create `public.gallery_moderation_events` with:

- `id`
- `submission_id`
- `moderator_id`
- `action`
- `reason`
- `created_at`

RLS will be enabled. Browser clients will not receive direct insert, update, or delete privileges on moderation events. `service_role` will manage moderation event rows through Edge Functions.
