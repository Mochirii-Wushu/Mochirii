## 1. Current Events Visual Audit

The Events page is functionally stable and already preserves the baseline behavior from the Events implementation and mobile accessibility passes. Current local smoke at 360px, 390px, 768px, and 1440px confirmed that the page loads, header and footer render, filters update in place, `aria-pressed` moves correctly, links keep their destinations and safe external attributes, and there is no horizontal overflow.

Event cards, filter controls, and schedule panels can carry a lantern-lit schedule identity without changing data, date parsing, sorting, copy, or renderer behavior.

## 2. Proposed Events Visual Treatment

Events should feel like a refined guild notice board and lantern-lit schedule: practical first, warm second. The visual system should guide members toward the next gathering, make filter state obvious, and keep past events useful without turning the page into a generic calendar app.

Accent palette:

- Primary: lantern gold for schedule emphasis, active states, and key rims.
- Secondary: ember for warmth around selected filters, featured surfaces, and event-card emphasis.
- Supporting: jade for trustworthy action links and small structural accents.
- Grounding: ink and warm glass for readable panels.
- Rare accent: moon for empty-state calm and quiet separators.

Panel and card tiers:

- Hero intro: featured schedule identity with static lantern/ember rim and restrained glow.
- Featured Event: warm feature panel with a clear image frame, date marker, title, summary, bullet rhythm, and link surface.
- Event Board: compact notice-board panel with tactile filters, a status count chip, event cards, and an intentional empty-state surface.
- Recurring Events: quiet support panel that reads as regular rhythm rather than a primary event.
- Participation: compact action/support panel with stable, tappable links.

How Events should differ from other visual systems:

- Different from Home: less threshold/doorway, more schedule board.
- Different from Gallery: less moonlit archive, warmer and more action-oriented.
- Different from Recruitment: not a long-form hall speech; denser and easier to scan.
- Different from Join: not an onboarding path; filters and dates are the main interaction.
- Different from Codex/Ranks/Leaders: less ceremonial hierarchy, more calendar clarity.
- Different from Side Pages: more active and time-aware than supporting bulletin pages.

Reusable visual-system lessons:

- Use `body[data-page="events"]` to scope all styling.
- Prefer static gradient rims over animated borders.
- Keep text and interaction behavior data-driven and unchanged.
- Preserve touch target sizes, visible focus, and no-overflow mobile layouts.
- Avoid JavaScript-driven visual effects.

Filter presentation:

- Keep the existing real buttons, labels, role group, and `aria-pressed` behavior.
- Style filters as a compact segmented schedule control.
- Make the selected filter visually stronger through lantern/ember fill and border.
- Keep inactive filters calm, readable, and obviously clickable.
- Add strong Events-scoped `:focus-visible` rings.

Event card presentation:

- Keep generated title, date/time/timezone line, summary, image, and link unchanged.
- Treat the meta line as a schedule marker.
- Give cards a stable rim, warm ink surface, and subtle hover/focus lift.
- Keep Past and All behavior visually clear without introducing new status labels.

Empty-state treatment:

- Preserve the existing empty-state text and filter-specific behavior.
- Present the empty state as a calm notice panel rather than a bare paragraph.
- Use a moon-supported, lantern-edged surface so the default empty Upcoming state feels intentional.

Motion and reduced motion:

- No new keyframes or JavaScript effects.
- Use only small transition-based hover/focus polish.
- Disable inherited Events panel animation by default.
- Existing reduced-motion safety remains compatible because the pilot does not rely on motion.

Must remain unchanged:

- `data/events.json`
- Event copy, dates, links, and rendered content
- Upcoming / Past / All behavior
- Default Upcoming filter
- UTC-safe date parsing and classification
- Sorting behavior
- Empty-state behavior and copy
- Header/footer behavior
- Protected content in Home, Recruitment, and Twills data
- Cache-query strings
