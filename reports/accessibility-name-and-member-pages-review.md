# Accessibility Name and Member Pages Review

Date: 2026-05-15
Branch: `qa/accessibility-name-and-member-pages-review`
Mode: QA/report-first

No site behavior, data, CSS, Supabase configuration, migrations, Edge Functions, assets, workflows, or protected content were changed for this review.

## 1. Scope

This review covers all current HTML pages plus the shared shell and member workflow scripts:

- Full pages: `account.html`, `announcements.html`, `auth.html`, `codex.html`, `events.html`, `gallery-submit.html`, `gallery.html`, `index.html`, `join.html`, `leader-dashboard.html`, `leaders.html`, `raffles.html`, `ranks.html`, `recruitment.html`, `spotify.html`, `spotlight.html`, `twills.html`.
- Shared fragments and scripts: `header.html`, `footer.html`, `site.js`, `styles.css`.
- Member workflow scripts: `auth.js`, `account.js`, `gallery-submit.js`, `leader-dashboard.js`.
- Prior roadmap basis: `reports/full-site-improvement-review.md`.

The review intentionally stayed report-only. It did not sign in, upload files, approve/reject submissions, mutate Supabase data, delete Storage objects, or run Supabase deploy/db commands.

## 2. Source Framework

Primary source: W3C WCAG 2.2, latest published recommendation. <https://www.w3.org/TR/WCAG22/>

Criteria used most directly:

- 1.1.1 Non-text Content: meaningful image alternatives and decorative image handling.
- 1.3.1 Info and Relationships: forms, lists, definition lists, and grouped controls must preserve structure.
- 2.1.1 Keyboard and 2.1.2 No Keyboard Trap: navigation, dialogs, dropdowns, and member controls must work without pointer input.
- 2.4.1 Bypass Blocks, 2.4.3 Focus Order, 2.4.4 Link Purpose, 2.4.6 Headings and Labels, 2.4.7 Focus Visible.
- 2.5.3 Label in Name and 2.5.8 Target Size (Minimum).
- 3.3.1 Error Identification and 3.3.2 Labels or Instructions.
- 4.1.2 Name, Role, Value and 4.1.3 Status Messages.

## 4. Accessible Name and Label Review

Strong evidence:

- Shared shell has a real skip link in `header.html` and every full page provides `#main`.
- Desktop nav dropdown buttons have visible text plus `aria-haspopup`, `aria-expanded`, and `aria-controls`.
- Mobile menu has `role="dialog"`, `aria-modal="true"`, a named close button, focus return, and Escape close behavior.
- Gallery and Home lightbox close buttons have explicit names.
- Member forms use wrapped labels for visible form controls:
  - Account profile fields in `account.html`.
  - Upload fields in `gallery-submit.html`.
  - Moderator decline reason textarea generated by `leader-dashboard.js`.
- Status/error regions are explicit on member pages:
  - `authStatus` / `authError`.
  - `uploadGuidance`, `verifyStatus`, `verifyError`, `profileStatus`, `profileError`, `accountSubmissionsError`.
  - `uploadStatus`, `uploadError`, `submissionsError`.
  - `reviewStatus`, `reviewError`.

Weakest accessible-name evidence:

- Home `#featuredBulletin` keeps `aria-label="Featured bulletin"` while rendered visible text becomes the actual event title and summary. Browser evidence after JSON render: visible text was `Event 02/28/2026 Monthly gatherings open to all members to be heard.`, but the accessible name remained `Featured bulletin`.
- Home `#spotlightCard` keeps `aria-label="Spotlight card"` while rendered visible text includes the current spotlight name and appreciation text. Browser evidence after JSON render: visible text included `Congratulations to: Meenari...`, but the accessible name remained `Spotlight card`.

Why this matters: WCAG 2.4.4 expects link purpose to be determinable, and WCAG 2.5.3 expects accessible names to include visible text for labeled controls. The current links are not unnamed, but their names are too generic once the JSON content loads.

## 5. Form Label Review

No missing visible form labels were found in the local heuristic pass.

Account page:

- `display_name`, `game_uid`, `discord_handle`, `region`, `timezone`, `avatar_url`, and `bio` are wrapped in `label.form-field`.
- Required display name has `required`, `minlength`, `maxlength`, and `autocomplete="nickname"`.
- URL field has `inputmode="url"` and `autocomplete="url"`.
- Save errors use `role="alert"`; save status uses visible text in a status paragraph.

Upload page:

- `imageFile`, `title`, `caption`, and `category` are wrapped in labels.
- Accepted file types and browser limit are visible near the form.
- Upload errors use `role="alert"`; upload progress/success uses a polite live status.

Leader dashboard:

- Queue filter buttons have visible labels and `aria-pressed`.
- Generated decline reason textarea is wrapped in a visible label.
- Approve/Decline buttons have visible text.

Limit: signed-in/verified/moderator-only states were not exercised with live credentials, so the review proves source structure and signed-out rendering, not every live positive path.

## 6. Button and Link Purpose Review

Strong evidence:

- Header/footer navigation links use clear visible text.
- Authentication buttons use action-oriented labels such as `Login with Discord`, `Sign out`, and `Check Discord verification`.
- Account action cards include visible command text plus short descriptive text inside the link/button.
- Gallery filters include visible category names and generated `aria-label`s with counts.
- Copy-link and sort controls have clear visible labels.

Issues to carry forward:

- `index.html` Home rich cards use generic static `aria-label`s for dynamic link content.
- Brand links in `header.html` and `footer.html` use `aria-label="Mōchirīī Home"` while the visible brand block also includes the subtitle. This is likely acceptable in practice because the primary visible brand text is included, but a strict label-in-name pass should decide whether to remove the `aria-label` or include the visible subtitle.
- Home Gallery spotlight links intentionally navigate to Gallery categories after rotation. The hidden Home lightbox shell remains in `index.html`; it is not exposed while hidden, but future audits should treat Home screenshot spotlight as category links unless product direction changes.

## 7. Status and Error Review

Strong evidence:

- Member pages consistently separate status text from alert/error text.
- Errors that require attention use `role="alert"` and are hidden until populated.
- Progress/success states commonly use `role="status"` or `aria-live="polite"`.
- Gallery share/copy and count states use polite live regions.
- Leader dashboard queue updates use `reviewStatus` and queue summary cards.

Gaps:

- Live positive-path status transitions remain credential-gated. Examples include successful Discord verification, profile save after real auth, successful upload submission, moderation approval/rejection, and queue reload after a real decision.
- The required validation ladder does not currently include a reusable status-message accessibility smoke.

## 8. Keyboard Order and Focus Review

Local browser checks found:

- Member page tab order starts at `Skip to main content`, proceeds through the brand/CTA/mobile menu, then page-specific signed-out controls, then footer navigation.
- Focus indicators are visible for the first keyboardable controls on `auth.html`, `account.html`, `gallery-submit.html`, and `leader-dashboard.html`.
- Mobile menu opens from the burger button, moves focus to `Close menu`, traps Tab within the menu, closes with Escape, sets `aria-expanded` back to `false`, restores `aria-hidden`, and returns focus to the burger.
- Desktop dropdown buttons open with ArrowDown, focus the first menu item, and close with Escape.
- Gallery lightbox opens from a Gallery thumbnail, moves focus to `#lightboxClose`, has a named dialog, and closes with Escape.

Home note:

- The Home screenshot spotlight currently rotates into category links. Clicking the rendered link navigates to `gallery.html?category=...`, so the Home hidden lightbox shell is not part of the active Home gallery interaction after rotation.

## 9. Focus Visibility Review

`styles.css` includes broad `:focus-visible` styles and component-specific focus states for:

- Global focusable controls.
- Skip link.
- Header, footer, mobile menu, CTA, burger, and icon buttons.
- Join, Events, Spotify, Gallery, Home, Recruitment, Twills, Codex/Ranks/Leaders, forms, account action cards, queue tabs, Gallery thumbs, and lightbox close buttons.

The browser keyboard pass observed solid `outline: 2px` focus indicators on member-page controls. A future axe/manual branch should add screenshot evidence for focus visibility at desktop and mobile widths, especially after any CSS refactor.

## 10. Mobile Target Review

The `390x844` heuristic pass did not find visible interactive targets below 24 by 24 CSS pixels on any full page. This aligns with WCAG 2.5.8 Target Size (Minimum), subject to manual review exceptions for inline text links and user-agent controls.

Controls checked included:

- Header mobile menu and close controls.
- Auth/account/upload/leader buttons and links.
- Gallery filter buttons, sort select, copy link, thumbnails, and lightbox close.
- Events filter buttons.
- Spotify search and filter chips.
- Footer navigation.

## 11. Signed-Out, Member, and Moderator State Notes

Signed-out states:

- `account.html` shows a login-required panel and hides account details.
- `gallery-submit.html` shows the upload gate and hides the upload form.
- `leader-dashboard.html` shows sign-in or access-denied panels and hides the review queue.

Member states:

- Account dashboard uses live status text for upload eligibility and verification guidance.
- Profile fields are labeled and disabled during busy states.
- My Gallery summary and recent-submission lists use live regions.

Moderator states:

- Queue tabs use buttons with `aria-pressed`; this is acceptable for filter buttons and avoids overclaiming tab semantics.
- Review queue uses a live list, status text, alert errors, and labeled decline reason.
- Storage reference is inside a `details` block with a visible `summary`, which keeps the low-level value discoverable without making it first-order screen-reader text.

Limitations:

- No live authenticated user, verified active member, or moderator account was used in this branch.
- No upload, approval, rejection, deletion, or Storage cleanup action was performed.
- No axe-core runner exists in the repo today.

## 12. Findings

### P0

No P0 accessibility failures were found in the report-only review.

### P1-1: Home dynamic card links have generic accessible names

Issue: `#featuredBulletin` and `#spotlightCard` keep static generic `aria-label`s after JSON content loads.

Where:

- `index.html` lines 125 and 162.
- `home.js` renders the visible text but does not update or remove the `aria-label`.

Why it matters: screen-reader and voice-control users may hear generic links instead of the loaded event/spotlight purpose. This affects WCAG 2.4.4 Link Purpose and WCAG 2.5.3 Label in Name.

Severity: P1.
Effort: Low.
Risk: Low.

Recommended branch: `fix/home-dynamic-card-accessible-names`

Expected files:

- `home.js`
- optional `index.html` if the static fallback labels are adjusted
- optional accessibility smoke script only if that branch is widened

Files that must not change:

- Protected content fields.
- `data/home.json` content unless explicitly requested.
- Supabase files.

Validation gate:

- Home browser check proves `#featuredBulletin` and `#spotlightCard` accessible names include or defer to visible loaded text.
- `npm run check`
- `git diff --check`
- `npm run check:production`
- `npm run smoke:gallery`

Definition of done:

- The accessible names for the two Home links are specific after data render.
- Visible card text, destination, and styling remain unchanged.

### P1-2: Accessibility checks are ad hoc, not durable validation

Issue: This branch used a one-off Playwright heuristic. The repo has no committed accessibility smoke script for accessible names, form labels, target sizes, dialog focus, mobile menu focus trap, or status regions.

Where:

- `package.json`
- `scripts/`
- Current validation ladder described in `reports/full-site-improvement-review.md`.

Why it matters: accessibility regressions are easy to reintroduce during CSS, Home, Gallery, or member workflow changes. WCAG criteria are testable, but the current required validation does not test them.

Severity: P1.
Effort: Medium.
Risk: Low-Medium.

Recommended branch: `qa/accessibility-smoke-script`

Expected files:

- `scripts/smoke-accessibility-basics.mjs`
- `package.json`
- optional `.github/workflows/*` only after CI cost is accepted

Files that must not change:

- Data files.
- Protected content.
- Supabase files.

Validation gate:

- New script passes locally and is documented as heuristic, not a WCAG conformance claim.
- Standard validation commands pass.

Definition of done:

- A reusable script covers page names, form labels, obvious empty names, target-size smoke, mobile menu focus trap, Gallery lightbox focus, and member signed-out gates.

### P1-3: Live member accessibility states are still credential-gated

Issue: Source and signed-out states are strong, but successful OAuth, Discord verification, active member upload, moderator queue review, approval/rejection, and cleanup states were not exercised with live test accounts.

Where:

- `auth.js`
- `account.js`
- `gallery-submit.js`
- `leader-dashboard.js`
- Supabase Edge Function-backed flows

Why it matters: WCAG 3.3.1, 3.3.2, and 4.1.3 depend on real status/error behavior during the complete process, not only static source structure.

Severity: P1.
Effort: Medium.
Risk: Medium because live tests can mutate data.

Recommended branch: `qa/member-accessibility-live-state-smoke`

Expected files:

- report first under `reports/`
- optional screenshots under a report evidence folder

Files that must not change:

- Supabase schema/config/migrations/functions.
- Data files.
- Production member records except approved test-account records under explicit cleanup rules.

Validation gate:

- Uses approved test accounts from `reports/member-workflow-test-account-matrix.md`.
- Does not upload or moderate real member content.
- Captures no tokens, secrets, signed URLs, or private user data.

Definition of done:

- Positive and failure states have screen-reader-visible status/error evidence and cleanup status.

## 14. Conclusion

The site has a good accessibility baseline for a static, JSON-rendered guild site: real skip links, one heading per page, clear visible labels, live status/error regions, focus-visible styling, working mobile menu focus management, and named Gallery dialogs. The weakest current point is not a catastrophic missing label; it is dynamic accessible-name drift where rich Home links keep generic labels after content loads, plus the lack of a durable accessibility smoke to keep future changes honest.

Best next branch after this report: `fix/home-dynamic-card-accessible-names`, because it is small, low-risk, user-facing, and directly grounded in WCAG link-purpose and label-in-name criteria.
